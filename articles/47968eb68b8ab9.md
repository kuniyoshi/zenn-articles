Unity でチーム開発しているときに、次のように実装がないプロパティを定義しているコードを見かけることがありました。

```C#
class Foo
{
    internal int Value { get; set; }
}
```

自分ではそのように書かないので「なんで無駄コードを書いているんだろう」と疑問に思います。つまり、下記のようにフィールドにすればいいんじゃないかということです。

```C#
class Foo
{
    internal int Value;
}
```

過激な思考では「フィールドだと罪悪感があるけど、プロパティなら許されるとでも思ってるんだろうか」とまで考えました。

あるいは「プロパティだと途中に処理を入れたくなっても簡単に変更できる」という主張が想像できます。しかし、フィールドにしていても同じように簡単にできます。設計としてプロパティとフィールドとに別の意味を持たせるパターンも想像できます。使い心地が同じなので意味の違いを維持するのは難しそうです。価値があれば難しくてもやるのがいいですが、大した価値は想像できません。

どうしてわざわざ冗長な書き方をするのかとずっと疑問だったわけですが、AI に聞くと .NET の文書にそう書いてあることがわかりました。

[Field Design](https://learn.microsoft.com/en-us/dotnet/standard/design-guidelines/field?utm_source=chatgpt.com)

確かに `DO NOT` となっています。しかしここには理由は書かれていません。もう一度 AI に聞くとプロパティだと、バイナリを更新しても利用側の再コンパイルが要らないとのことです。でも「依存ライブラリを更新しておいてコンパイルしない場合っていつ?」となりました。AI に聞き込みを付けました。そして「アプリケーション間で共通して使う DLL だとあり得そうだ」とわかりました。さらに聞き込みを続けると、昔は OS の DLL を共用する設計だったらしいです。リソースが増えてきたので節約するメリットとデメリットのバランスが変わるやつですね。

というわけで自分が書く範囲のプログラムではプロパティにする恩恵を受けることはなさそうです。とはいえ特にどっちでもいいと思うのでプロジェクトの雰囲気に合わせるのが良さそうです。
